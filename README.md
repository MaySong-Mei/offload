# Offload

**Offload turns your phone into a remote control for the coding agents running on your dev machine.**

The core idea: each repo on your dev machine gets a `.offload/` directory that acts as the interface between the human (via phone) and a local coding agent (e.g., Claude Code CLI). You write instructions on your phone → those instructions land as files inside `<repo>/.offload/` → the local agent reads them, works on the real codebase, writes results back. Your phone is the controller; the dev machine is the workshop; the agent is the worker.

## How it works

```
┌──────────────┐                ┌────────────────────────────────────┐
│   iPhone     │ ──HTTP/WS────► │  Offload Server (on dev machine)   │
│  (controller)│                │  • SQLite index (cross-repo query) │
└──────────────┘                │  • File proxy for .offload/        │
                                │  • Launches Claude in repo dir     │
                                └────────────┬───────────────────────┘
                                             │ reads/writes
                                             ▼
                                ┌─────────────────────────────────┐
                                │  Repo (your real codebase)      │
                                │  ├── src/                       │
                                │  ├── .gitignore  ── /.offload/  │
                                │  └── .offload/   ◄── source of  │
                                │      ├── init/        truth     │
                                │      ├── context/               │
                                │      ├── topics/                │
                                │      └── state.json             │
                                └─────────────────────────────────┘
                                             ▲
                                             │ runs in repo dir,
                                             │ reads .offload/,
                                             │ touches real files
                                ┌────────────┴─────────┐
                                │  Local Claude CLI    │
                                └──────────────────────┘
```

### `.offload/` layout

The `.offload/` directory inside each managed repo is the single source of truth for everything Offload knows about that repo:

```
.offload/
├── init/                          # Onboarding template (copied from offload server on first init)
│   ├── INIT_PROMPT.md             # Instructions for Claude: "read this repo, fill out context/"
│   ├── README.md                  # Explains what .offload/ is, for humans
│   └── templates/                 # Skeleton files Claude fills in during init
│       ├── summary.template.md
│       ├── architecture.template.md
│       ├── conventions.template.md
│       └── glossary.template.md
│
├── context/                       # Generated by Claude during onboarding (the repo's "long-term memory")
│   ├── summary.md                 # One-paragraph summary; shown on iOS project cards
│   ├── architecture.md            # High-level module layout, key abstractions
│   ├── conventions.md             # Coding style, idioms, gotchas
│   └── glossary.md                # Domain terms
│
├── topics/                        # Each topic = one work item (feature, bug, refactor)
│   └── topic-<id>/
│       ├── topic.md               # Original idea dump
│       ├── requirement.md         # Refined requirement
│       ├── plan.md                # Implementation plan
│       ├── notes.md               # Side notes
│       ├── state.json             # State machine (requirement/execution/decision)
│       ├── feedback/fr-<id>.json  # Pending feedback requests
│       └── runs/run-<id>/         # Each agent execution
│           ├── stdout.log
│           ├── stderr.log
│           └── result.json
│
└── state.json                     # Repo-level metadata (init version, last init time)
```

**By default the entire `.offload/` directory is gitignored.** Add Offload to your repo's `.gitignore` (the server does this for you on first init) and your private instructions stay local. If you want to share the long-term context (`context/`) with your team, manually un-ignore that subdirectory in `.gitignore`.

### Lifecycle

1. **Discover.** The server scans `--projects-root` (default `~/code`) for git repos. Each becomes a "project."
2. **Initialize.** From the iPhone, you tap "Initialize" on a project card. The server copies the onboarding template into `<repo>/.offload/init/`, adds `.offload/` to `.gitignore`, then runs Claude in the repo with `INIT_PROMPT.md` as the prompt. Claude reads the repo and fills out `<repo>/.offload/context/*.md`.
3. **Create topics.** From the phone you write a topic (raw idea + tags). The server writes it to `<repo>/.offload/topics/<id>/`. State machine starts in `captured`.
4. **Refine.** Approve the requirement → approve the plan. Each step triggers Claude to refine the docs in `.offload/topics/<id>/`.
5. **Launch agent.** Tap "Launch Agent." The server spawns Claude in the repo directory, with `.offload/context/*.md` + the topic's `requirement.md` + `plan.md` as the prompt. Claude does real work in the repo. Stdout/stderr/diffs are written to `runs/<run-id>/`.
6. **Review.** On the phone you see runs, status, and any feedback requests Claude raised. Approve, request changes, or trigger another run.
7. **Archive.** When the topic is done, you mark it passed and archive it. The `.offload/topics/<id>/` directory remains as the audit trail.

## Components

| | Path | Purpose |
|---|---|---|
| Server | `server/offload/` | Python HTTP/WS server, file proxy, Claude executor, SQLite index |
| iOS client | `clients/ios/OffloadClient/` | SwiftUI app: project cards → topics → details, offline-capable |
| Onboarding template | `server/offload/templates/repo-init/` | Files copied into `<repo>/.offload/init/` on first init |
| Tests | `tests/` | Server unit tests; iOS test target lives next to the Xcode project |

## Server

Run from the repo root (zero third-party dependencies, just Python 3.9+):

```bash
python3 -m server.offload \
  --workspace /tmp/offload-workspace \
  --host 127.0.0.1 \
  --port 8080 \
  --projects-root ~/code
```

Optional bearer token authentication:

```bash
OFFLOAD_API_TOKEN=dev-token python3 -m server.offload --workspace /tmp/offload-workspace
```

The `--workspace` directory only holds the SQLite index now; topic data lives inside each repo's `.offload/`.

## iOS client

Open `clients/ios/OffloadClient/OffloadClient.xcodeproj` in Xcode and run on a simulator or device. The app needs the server URL (default `http://127.0.0.1:8080`); for a real device, point it at your dev machine's address (a Tailscale IP, MagicDNS hostname, or LAN IP all work).

The iOS app supports offline editing: when disconnected, mutations are queued locally and replayed when the connection is restored.

## Self-Iteration: April 14, 2026

On this date, Offload successfully used itself to modify its own codebase — the first self-bootstrapping iteration.

**What happened:**

1. A topic was created via the Offload API: *"Simplify project dashboard — move pipeline and activity into meta card"*
2. The Offload agent (Claude Code) read the Offload iOS client source code (`Views.swift`), identified the exact lines to change, and wrote a structured `requirement.md` with specific file paths and line numbers
3. After human approval on the phone, the agent generated a detailed 8-step `plan.md`
4. After human approval, the agent executed the plan: deleted 184 lines of dead code, added 54 lines of new code, enriched the Meta Card with pipeline stats and agent activity indicators, and added attention badges to topic rows
5. The agent wrote a structured report. The human reviewed and pushed the commit: `ed49fe0`

**Why this matters:**

This proved that the Offload architecture works end-to-end — a human on a phone can describe a change, an agent understands the codebase, plans the work, executes it, and reports back. The `.offload/` data object served as the single source of truth between the agent sessions and the human gates. No manual coding was needed; the system iterated on itself.

The commit message reads: *"feat: simplify dashboard — pipeline + activity in meta card (offload self-iteration)"*

## Status

Active development. The core architecture — `.offload/` data objects, agent-powered clarification and planning, human confirmation gates, streaming session output to iOS — is functional. The system can iterate on itself.

## Why "Offload"?

Because the work gets offloaded — from your head to a topic, from your laptop to the dev machine, from you to the agent. You stay on your phone, in the loop, in control.
